NORTHEPTION 動画ショート化ツール 仕様書

作成日：2026年6月16日（v1）／最終更新：2026年6月17日（v7）／用途：オバクック等の長尺ゲームプレイ動画から、ショート候補を半自動生成する

v7での変更点（実装後の反映・v6の追加対策）： v6のVADフィルタだけではハルシネーションを防ぎきれない事例（キリル文字や英単語の混入）が別の候補でも見つかったため、追加対策としてクリップ開始直前の実際の発言（動画全体の文字起こしから取得）をfaster-whisperのinitial_promptとして与えるようにした。文脈ゼロで始まることがハルシネーションの主因のため、本物の直前文脈を渡すことでこれを直接緩和する。 詳細は「10. クリップ単体文字起こしのハルシネーション対策」を参照。
v6での変更点（実装後の反映）： クリップ単体の再文字起こしで、無音/曖昧な音区間にWhisperが存在しない発言を作り出すハルシネーション（稀に韓国語など別言語の文字が混入することも）が発生する事例を確認した。原因は、動画全体を一括で文字起こしする際は前後の文脈（既に認識済みの日本語）が蓄積されているのに対し、クリップ単体の再文字起こしは文脈ゼロで始まるため、無音区間に当たると尤もらしい文字列を作ってしまうこと。対策としてVAD（無音区間検出）フィルタを有効化し、無音区間を文字起こし対象から除外するようにした。完全な防止はできないが、検証した事例では誤言語の混入が解消し、実際の発言内容に近い結果になった。 詳細は「10. クリップ単体文字起こしのハルシネーション対策（v6で追加）」を参照。
v5での変更点（実装後の反映・v4の修正）： v4で全クリップに適用した文字単位タイミング表示（カラオケ風）は、全文に適用すると逆に見づらいと判明したため、既定（自動生成時）は元の「一文ずつ表示」に戻した。代わりに、人間が見て「この箇所だけカラオケ風にしたい」と指定した区間だけを後から個別にカラオケ表示へ変換できるnpm run karaokeコマンドを追加した。 詳細は「9. 字幕の文字単位タイミング表示（v4で追加・v5で選択適用に変更）」を参照。
v3での変更点（実装後の反映）： ①Claude APIモードのreason（選定理由）が、クリップ切り出し後の実際の字幕内容から再生成されるようになり、文字起こし全体から推測した理由とのズレを解消。 ②候補が重複しないよう重複区間の自動排除を追加し、歩留まり確保のため要求件数を多めに投げるよう調整。 ③人間が動画を見て見つけた区間を、文字起こしの訂正と同時に手動指定でクリップ化できる機能（manual-clip）を追加。 詳細は「7-2. Claude APIモードの精度改善（v3）」「8. 手動クリップ指定機能（v3で追加）」を参照。
v2での変更点（実装後の反映）： ハイライト検出を音声ラウドネス方式とClaude API方式の2モードに対応（--mode loudness / --mode claude、既定はclaude）。 v1で「将来の拡張」としていたClaude API版を正式実装し、既定の検出方式に格上げした。旧方式（ラウドネス）は無料・APIキー不要の選択肢として残している。 詳細は「7. ハイライト検出：Claude APIモード（v2で追加）」を参照。

設計思想：人間の仕事は「判断」だけ

このツールは全自動投稿ツールではない。SNS botと同じ「1対多（機械が候補を出す・最終判断は人）」の動画版。機械が雑務（探す・切る・縦化・字幕）を全部やり、人間に残すのは「どれが良いか選ぶ」判断だけにする。

Before（手作業）After（このツール導入後）
1時間の動画をスクラブして良い瞬間を探す → 手で切る → 縦にリサイズ → 字幕を打つ → 結局やらない 動画を1本フォルダに置いてコマンド1つ → 縦化・字幕済みの候補が6〜8本出てくる → 短い候補を見て1本選ぶだけ

人間がやること（これだけ）

  1. 選ぶ：出力された候補クリップ（各15〜30秒・縦・字幕付き）を見て、出す1本を選ぶ
  2. 承認：キャプション（SNS botで生成）を確認し、双子が映る場合は保護者とすり合わせた上で手動投稿
探す・切る・縦化・字幕は全部ツールが自動でやる。人間の所要は「短い候補を数本見て選ぶ」＝数分。

全体フロー

① 1時間の動画（input.mp4）を /input フォルダに置く
② コマンド実行（例：npm run clips。既定は--mode claude）
③ ハイライト検出（モードにより分岐）
　・--mode claude（既定）：動画全体をWhisperで文字起こし → タイムスタンプ付きテキストをClaude APIに渡し、盛り上がり区間をN個選んでもらう
　・--mode loudness：音声ピーク（声・笑い）の時間帯を自動抽出 → 候補タイムスタンプN個
④ 各候補を FFmpegで切り出し（15〜30秒）
⑤ 9:16縦化（背景ぼかしレターボックス）
⑥ 各クリップをfaster-whisperで個別に文字起こしして字幕生成 → クリップに焼き込み
⑦ /candidates に clip_01.mp4 … clip_0N.mp4 ＋ manifest.json を出力
⑧ 人間が候補を見て1本選ぶ（以降は手動投稿）

技術スタック

項目内容
実行環境ローカルPC（Windows）。動画処理は重くFly.io無料枠に不向きなため、ローカルでバッチ実行する。SNS botは従来通りFly.io。
オーケストレーションNode.js（既存スタックに合わせる）。各CLIを child_process で呼び出す
動画処理FFmpeg（カット・縦化・字幕焼き込み・音声解析）。オープンソース・無料
文字起こし実装はfaster-whisper（pip）。当初予定のwhisper.cppはWindowsビルドが大変なため、仕様書記載の代替案として採用した
ハイライト検出2モード対応（--modeで切替、既定claude）
・claude：全体文字起こし＋Claude API（claude-sonnet-4-6）で盛り上がり区間を判定。精度優先
・loudness：音声ラウドネス時系列のピーク検出。完全無料・ローカル・APIキー不要だが誤検出が出やすい
コスト--mode claude（既定）は実行ごとに少額のAPI課金が発生（数円規模）。--mode loudnessなら0円

各ステップの実装指示

1. 入力

  • /input/input.mp4 を処理対象とする（ファイル名は引数で上書き可）
  • 作業用に音声だけ抽出：ffmpeg -i input.mp4 -ac 1 -ar 16000 -vn work/audio.wav（whisper.cpp用に16kHzモノラル）

2. 音声ピーク検出（候補タイムスタンプの抽出）（--mode loudness 時のみ。既定は8番のClaude APIモード）

盛り上がりの代理指標としてラウドネス（音量）の時系列を使う。掛け合いゲーム（オバクック2人プレイ）では声が上がる瞬間＝面白い瞬間と相関しやすい。

  • FFmpegの ebur128 フィルタで瞬時ラウドネス（Momentary, M値）を出力し、stderrから時刻＋M値をパースして1秒粒度の時系列を作る
    　例：ffmpeg -i work/audio.wav -af ebur128=metadata=1 -f null - 2> loudness.log → ログを正規表現でパース
  • 移動平均でノイズを均し、上位ピークを検出。近接するピークは1つの区間にマージ
  • 各ピークを中心に 15〜30秒の窓に広げる（前2〜3秒・後ろ長めにして"オチ"を含める）。窓が重なる候補は統合
  • ラウドネススコア上位から N件（既定8件）を候補として採用
限界（重要）：音声ピークは「うるさい＝面白い」ではない。ゲームSEや苛立ちの声も拾う。誤検出が出る前提で候補を多めに出し、最終選択は人間が行う（＝このツールの設計思想そのもの）。

3. クリップ切り出し（FFmpeg）

ffmpeg -ss [開始秒] -i input.mp4 -t [長さ秒] -c:v libx264 -c:a aac work/cut_NN.mp4

※ -ss を入力前に置くと高速シーク。フレーム精度が必要なら入力後に置く。

4. 9:16縦化（背景ぼかしレターボックス）

オバクックは俯瞰固定画面で、単純な中央クロップだと厨房の端が切れる。元映像をぼかして背景に敷き、その上に元映像全体を載せる方式が安全。出力は1080×1920。

ffmpeg -i work/cut_NN.mp4 -filter_complex \
"[0:v]scale=1080:1920:force_original_aspect_ratio=increase,crop=1080:1920,boxblur=20:5[bg]; \
 [0:v]scale=1080:-1[fg]; \
 [bg][fg]overlay=(W-w)/2:(H-h)/2" \
-c:a copy work/vert_NN.mp4

5. 字幕生成・焼き込み

  • 各クリップの音声を whisper.cpp に渡し .srt を生成（日本語モデル：ggml-medium 目安。精度優先なら large、速度優先なら small）
  • SRTをFFmpegで焼き込み：
    ffmpeg -i work/vert_NN.mp4 -vf "subtitles=work/sub_NN.srt:force_style='Fontsize=18,Outline=2,Alignment=2'" candidates/clip_NN.mp4
  • 字幕スタイルは下部中央・縁取り付き（音声オフ視聴対策）。フォントはNoto Sans JP等の日本語対応を指定
限界：ゲームSE混じりの音声では字幕に誤変換が出ることがある。人間は字幕が綺麗な候補を選べばよい（or 採用1本だけ手直し）。

6. 出力

  • /candidates/clip_01.mp4 … clip_0N.mp4（縦・字幕済み・即視聴可）
  • /candidates/manifest.json：各候補の { ファイル名, 元動画の開始/終了秒, 長さ, ラウドネススコア, 字幕テキスト }。人間が一覧で見て選びやすくするため

7. ハイライト検出：Claude APIモード（v2で追加・既定）

ラウドネス検出（うるさい＝面白い、という乱暴な代理指標）の誤検出を減らすため、文字起こしの内容そのものをClaudeに判断させる方式。仕様書v1の「将来の拡張」案を正式実装し、既定モードに格上げした。

  • 動画全体の音声を抽出し、faster-whisperで全文文字起こし（タイムスタンプ付き）
  • 文字起こしテキストを「[開始-終了] テキスト」形式に整形し、Claude API（モデル：claude-sonnet-4-6）に渡す
  • プロンプト例：「以下はゲーム実況の文字起こしです。2人の掛け合い・笑い・驚き・盛り上がりが伝わる区間をN箇所選び、各区間の開始秒・終了秒・選んだ理由をJSONで返してください」
  • 返ってきたJSON（[{"startSec":..,"endSec":..,"reason":".."}, ...]）のタイムスタンプを使い、以降は3〜6番の手順（切り出し・縦化・字幕焼き込み）を共通実行
  • 選定理由（reason）は manifest.json に格納し、人間が候補を選ぶ際の参考情報にする
注意：このモードはClaude APIを呼ぶため、APIキー（ANTHROPIC_API_KEY）が必要かつ実行ごとに少額の課金が発生する（v1で「完全無料」としていた前提から外れる）。APIキーはプロジェクト直下の .env ファイル（ANTHROPIC_API_KEY=sk-ant-...）で管理し、OneDrive経由で開発環境間を引き継ぐ。Windowsレジストリ等PC単体にしか効かない保存方法は使わない。課金やAPIキー設定を避けたい場合は --mode loudness（v1のラウドネス方式）に切り替えれば従来通り無料・ローカルのみで動作する。

7-2. Claude APIモードの精度改善（v3で追加）

運用してみると、Claudeが返すreason（選定理由）が、実際に切り出した区間の発言と微妙にズレる事象があった（文字起こし全体を見て判断するため、近接する別タイミングの発言を理由に混ぜてしまう）。これを解消するため、以下を実装：

  • reasonの再生成：クリップ切り出し→縦化→個別文字起こし（既存の手順5）が終わった実際の字幕テキストだけを根拠に、もう一度Claude APIを呼んでreasonを短文（40字程度）で書き直す（src/claude-detect.js の summarizeClipReason()）。最初の選定時に使ったreasonは内部的にしか使わず、最終的にmanifest.jsonへ書き出すのは再生成後のものになる。
  • 重複区間の自動排除：Claudeには「区間は重複しないように」と指示しているが、必ずしも守られない。開始秒でソートした上で、前に採用した区間と重なる区間をgreedyに除外する（clampAndValidate()）。
  • 歩留まり確保：重複排除で候補が減る前提で、実際はcandidateCount + 4件をClaudeにリクエストしてから絞り込む。

8. 手動クリップ指定機能（v3で追加）

Claude APIによる自動検出は文字起こしの精度に依存するため、人間が実際に動画を見て見つけた「ここが面白い」という区間を直接指定したいケースがある。さらに、自動文字起こしの誤変換を人間が訂正したいケースもある。この2つを同時に解決する機能。

使い方

  1. 自動検出（npm run clips）を先に実行し、work/full_transcript.srt（動画全体のタイムスタンプ付き文字起こし）を生成しておく
  2. 人間が動画を見て気になった区間を、full_transcript.srtから該当箇所をコピーし、セリフの誤りを訂正する
  3. 複数区間をまとめて指定したい場合は、ブロックの間に===の行を入れて区切る
  4. 訂正済みテキストをファイルに保存し、以下を実行： 
    npm run manual-clip <貼り付けたテキストのファイルパス>

処理内容

  • 貼り付けられたテキストから、タイムスタンプ行（HH:MM:SS,mmm --> HH:MM:SS,mmm）を起点にエントリを抽出する（src/manual-clip.js の parseManualBlocks()）。人間が編集するのはセリフ部分のみで、タイムスタンプ・連番行は元のfull_transcript.srtのまま使う想定
  • 各ブロックの最初のエントリの開始秒・最後のエントリの終了秒を、そのクリップのstartSec・endSecとして採用
  • 各エントリのタイムスタンプをクリップ先頭基準（0秒始まり）に変換し、訂正済みテキストでクリップ用SRTを再構築する（Whisperによる再文字起こしは行わない＝誤変換が再発しない）
  • 既存パイプラインと同じ cutClip（切り出し）→ toVertical（縦化）を実行し、上記で作った訂正済みSRTをそのまま字幕焼き込みに使う
  • reasonは、訂正済みの字幕テキストを根拠にClaude APIで生成する（7-2のsummarizeClipReason()を再利用）
  • 生成したクリップは既存のcandidates/manifest.jsonに追記される（index番号は既存候補の続きから採番。自動検出分は上書きしない）
この機能は既存の自動検出パイプライン（npm run clips）を置き換えるものではなく、追加の手段。「まず自動で候補を作ってもらい、出来を見て必要な分だけ人間が手動で補う」という運用を想定している。

9. 字幕の文字単位タイミング表示（v4で追加・v5で選択適用に変更）

従来は faster-whisper のセグメント（文・フレーズ単位）の開始時刻に、そのセグメントの全文が一度に表示されていた。「ちょちょちょちょ」のような早口の反復発言では、実際は0.5秒に1回ずつ言っているにもかかわらず、字幕は最初の発話タイミングで全文が一括表示されてしまい、見た目の臨場感に欠ける問題があった。

v4では文字単位タイミング表示（通称カラオケ表示）を全クリップ・全文に適用したが、運用してみると通常の発言まで1文字ずつ増えていく表示になり逆に読みづらいと判明した。そのため v5 では既定の動作を「一文ずつ表示」に戻し、カラオケ表示は人間が指定した区間にだけ後から適用する方式に変更した。

9-1. 文字単位タイミングの仕組み（共通ロジック）

  • faster-whisperの word_timestamps=True を有効化し、単語単位の開始時刻を取得する
  • 単語の終了時刻はアライメント精度が低く信頼できない（0秒や不自然に長い区間になることがある）ため使用しない。代わりに各単語の表示区間は「次の単語が開始する時刻まで」とする
  • 各単語の表示区間を、その単語の文字数で均等に分割し、文字ごとの出現タイミングを算出する
  • 文字を1つずつ追加していく累積テキストでSRTエントリを構築する（カラオケ字幕に近い見え方）。出現タイミングが同時（0秒区間）になった文字は、次の有効なタイミングのエントリにまとめて吸収される（不自然なチラつきを防止）
  • セグメント（文）の境目では累積テキストをリセットする（次の文は空から積み上がる）
限界：faster-whisperの単語アライメントは、高速な反復音節（「ちょちょちょちょ」等）では精度が粗く、一部の文字が1〜2秒程度まとめて表示される箇所が残る。完全に発話タイミングと一致するわけではないが、従来の「セグメント全文を一括表示」よりは大幅に改善される。

9-2. 既定の動作（v5）

npm run clips・npm run manual-clip による字幕生成（work/sub_NN.srt）はすべて元の「一文ずつ表示」に戻した。動画全体の文字起こし（work/full_transcript.srt）も従来通り文単位のまま変更していない。

9-3. 選択的カラオケ化（npm run karaoke、v5で追加）

人間がクリップを見て「この区間だけカラオケ風にしたい」と判断した部分だけ、後から個別に変換できるコマンド。2つのモードがある。

モードA：範囲指定（自動で文字数に応じて均等割り）

npm run karaoke -- <candidate番号> <開始秒-終了秒> [<開始秒-終了秒> ...]
例: npm run karaoke -- 3 12.5-14.0 20.0-21.2
  • 指定した区間と時間的に重なるセグメント（文）だけが文字単位タイミング表示になり、それ以外は一文ずつ表示のまま維持される（python/transcribe.pyのgranularity=mixed）
  • 文字の出現タイミングはfaster-whisperの単語タイムスタンプから自動算出（9-1のロジック）。手軽だが、反復音節などではタイミングが粗くなる限界がある

モードB：タイミングファイル指定（文字ごとに出現時刻を厳密指定、v5で追加）

9-1の自動算出では精度が粗い場面向けに、文字ごとの出現時刻を人間が直接指定できるモード。0.1秒単位での指定を想定。

npm run karaoke -- <candidate番号> <タイミングファイルパス>

タイミングファイルの形式（1行＝「時刻 表示する文字（複数文字も可）」、最後の行は時刻のみで表示終了タイミングを示す）：

12.3 あ
12.5 い
12.8 う
13.0 え
13.4 お
13.8
  • 動画を見ながら文字ごとの実際の発話タイミングを0.1秒単位で書き出し、ファイルに保存して渡す
  • 複数区間を一度に指定する場合は、ブロックの間に===の行を入れる
  • 指定した時間範囲（最初の行の時刻〜最後の行の時刻）に重なる既存の字幕エントリは削除され、指定したタイミングのエントリに置き換わる。範囲外の字幕は変更されない
  • このモードはfaster-whisperの再実行を行わない（既存のwork/sub_NN.srtを直接書き換えるだけ）ため高速

共通の注意点

  • 秒数は対象クリップ内のローカル時間（candidates/clip_NN.mp4を再生して見た時刻、またはwork/sub_NN.srtの時刻）を指定する
  • 実行するとwork/sub_NN.srtを更新し、candidates/clip_NN.mp4を字幕焼き込みからやり直す（対象クリップのwork/vert_NN.mp4が既に存在している必要がある＝先にnpm run clips等で生成済みであること）
  • モードAは実行ごとにクリップ全体を最初から再生成する。モードBは既存のSRTに対して指定範囲だけを差し替える

10. クリップ単体文字起こしのハルシネーション対策（v6で追加・v7で追加対策）

運用中、ある候補（37秒のクリップ）で、字幕とそれを根拠にしたreasonが動画全体の文字起こし結果と全く食い違う事例が見つかった。クリップ単体の再文字起こし結果には韓国語の文字まで混入していた。

原因

  • このクリップの実際の音声は、約37秒のうち半分近くが無音・料理SEのみの区間だった
  • 動画全体を一括で文字起こしする場合：faster-whisperは既定で前のセグメントの文脈を踏まえて次を予測する（condition_on_previous_text）。無音区間に差し掛かった時点で、既に数十分分の正しい日本語の文脈が蓄積されているため「次も日本語の会話の続きだろう」と正しく推測しやすい
  • クリップ単体で再文字起こしする場合：そのクリップの先頭から文脈ゼロで始まる。無音や曖昧な音に当たると、Whisperは「何か言っているはず」と推測モードに入り、学習データ中の尤もらしい文字列を作り出してしまう（ハルシネーション）。稀に言語そのものを取り違えて別言語の文字を生成することもある（language="ja"を強制指定していても発生し得る、Whisperの既知の弱点）

対策：VADフィルタの有効化

  • faster-whisperのvad_filter=Trueを全ての文字起こし呼び出し（python/transcribe.py）に有効化した。文字起こし前に音声区間検出（VAD）を行い、無音・非発話区間を最初から文字起こし対象から除外する
  • 検証した事例では、有効化前は韓国語混入を含む全く無関係な内容になっていたが、有効化後は誤言語の混入が解消し、実際の発言（「サラがない」「やばいやばいやばい」等）に近い結果が得られた
限界：VADフィルタはハルシネーションを完全には防げない。無音区間の判定自体も完璧ではなく、短い曖昧な発話区間ではなお誤った文字列が生成されることがある。実際、VAD導入後も別の候補でキリル文字や英単語が混入する事例が見つかった。

追加対策：initial_promptによる文脈の付与（v7で追加）

VADだけでは不十分だったため、ハルシネーションの根本原因（クリップ単体の再文字起こしが文脈ゼロで始まること）に直接対処する追加策を実装した。

  • 動画全体の文字起こし（Claude APIへの候補検出時に既に取得済みのsegments）から、各クリップの開始時刻（startSec）の直前にある実際の発言を最大120文字程度抜き出す（src/claude-detect.jsのbuildInitialPromptText()）
  • これをfaster-whisperのinitial_prompt引数として渡す（python/transcribe.pyに--initial-prompt TEXTオプションを追加）。動画全体の文字起こしが文脈の蓄積によって精度が高いのと同じ理屈で、クリップ単体の再文字起こしにも本物の直前文脈を与えることで、デコーダーが「日本語の会話の続き」だと正しく推測しやすくなる
  • このオプションは--mode claudeでのみ使用可能（動画全体の文字起こし結果が必要なため）。--mode loudnessでは従来通り文脈なしで実行される
限界（再掲）：VAD＋initial_promptの組み合わせでも、ハルシネーションを完全には防げない。字幕・reasonに明らかな違和感がある場合は、work/full_transcript.srt（動画全体の文字起こしは文脈があるため精度が高い）と見比べて確認し、必要なら手動クリップ指定機能（8番）やカラオケのタイミングファイル指定（9-3モードB）で訂正することを推奨する。

前提・初回セットアップ（一度だけ）

セットアップ自体もClaude Codeに実行させる。以降は「動画を置く→コマンド」だけ。

  • FFmpeg をインストールしPATHを通す
  • whisper.cpp をビルド（or プレビルドバイナリ）＋ 日本語対応 ggml モデル（medium）をダウンロード。Windowsでビルドが詰まる場合は pip install faster-whisper に切替
  • Node.js（既存環境）
  • 日本語フォント（字幕焼き込み用）
PC性能の確認：Whisperのローカル文字起こしはCPU/GPUに依存。1時間動画の全文字起こしは重いが、本ツールは切り出した短いクリップ（合計数分）だけを文字起こしするので負荷は小さい。初回に1クリップで所要時間を計測し、モデルサイズを調整する。

正直な限界と、それを判断で吸収する設計

  • （--mode loudness時）音声ピーク検出は「面白い」を保証しない（うるさい瞬間を拾う）。→ 候補を多め＋人間が選ぶことで吸収
  • （--mode claude時）Claudeの判断は文字起こし（faster-whisper）の精度に依存する。文字起こし自体が誤っていれば、Claudeもその誤った内容を基に判断する。また実行ごとに少額の課金が発生し「完全無料」ではなくなる
  • 字幕は誤変換が出る。→ 綺麗な候補を選ぶか採用1本だけ手直し
  • クラウドで「全自動・無料・透かし無し」は実在しない。無料を貫くなら--mode loudness＋人の最終確認がこのツールの条件
